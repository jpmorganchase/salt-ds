# Proposal — migrate `@salt-ds/ag-grid-theme` to the AG Grid Theming API (v33+)

**Status:** Draft for discussion
**Owner:** TBD
**Target package version:** `@salt-ds/ag-grid-theme@3.0.0`
**AG Grid versions in scope:** v33.x, v34.x (Theming API). v32 stays on the existing 2.x line.

> Research sources: <https://www.ag-grid.com/javascript-data-grid/theming/> and the linked sub-pages: `theming-parameters`, `theming-parts`, `theming-colors`, `theming-borders`, `theming-compactness`, `theming-css`, `theming-distribution`, `theming-migration`.

---

## 1. TL;DR

AG Grid v33 introduced a new **Theming API**. Themes are no longer CSS files added via a `class="ag-theme-*"` wrapper — they are JavaScript objects (`Theme` + `Part`s) passed to the grid via the `theme` grid option. The legacy CSS-class system is **deprecated** and will be removed in a future major version.

We should ship a new major (`3.0.0`) of `@salt-ds/ag-grid-theme` that:

- Exports a `saltTheme` (JS object) built with `createTheme()` + `withPart()` + `withParams()`.
- Ships Salt-specific styling as **typed parts** (`iconSet`, `checkboxStyle`, `inputStyle`, `columnDropStyle`, header parts) plus three small no-feature parts (`saltCellStates`, `saltFocusRing`, `saltRangeSelectionAdjustments`) for the residual bespoke CSS.
- Replaces the four hard-coded class selectors per rule (`ag-theme-salt-{light,dark,compact-light,compact-dark}`) by relying on **inherited** `--salt-*` tokens for both mode and density — no `data-ag-theme-mode` wiring required by default.
- Keeps the legacy 2.x line for users still on AG Grid v32 / `provideGlobalGridOptions({ theme: "legacy" })`.

Estimated effort: **~13–17 engineering days** plus a ≥ 4-week pre-release soak.

### 2.1 What lands on the 2.x line before this migration

The branch `fix/ag-grid-theme-align-with-table` (changeset `ag-grid-theme-header-variants-and-pill-tokens.md`, package bump: minor) ships the following ahead of v3. All five items are **prerequisites for v3** — each one removes friction for the migration:

- **Header background and divider modifier classes** (`ag-theme-salt-header-{primary,secondary,tertiary}`, `ag-theme-salt-header-divider-{primary,secondary,tertiary,none}`). Mirror Salt `Table`'s `<THead variant divider />` API. See issue [#5391](https://github.com/jpmorganchase/salt-ds/issues/5391) and the new `css/parts/ag-header-variants.css` + `src/examples/HeaderVariants.tsx`. In v3 these become typed parts — see §4.6.1.
- **Row Group Panel pill tokens** aligned with Salt `Pill`. `--ag-chip-background-color` in `ag-root-var.css` is now `--salt-actionable-bold-background`, and `ag-column-drop-list.css` cascades `--ag-secondary-foreground-color`, `--ag-icon-font-color` and `--ag-chip-background-color` across the default / `:hover` / `:active` / `[aria-expanded="true"]` states using the matching `--salt-actionable-bold-*` tokens. See issue [#6196](https://github.com/jpmorganchase/salt-ds/issues/6196). In v3 this becomes the `saltColumnDropStyle` part — see §4.6.2.
- **Deprecated Salt token cleanup.** Replaced `--salt-actionable-secondary-*` (renamed to `--salt-actionable-subtle-*`) and `--salt-actionable-cta-*` (renamed to `--salt-actionable-accented-bold-*`) inside `ag-buttons.css` and `ag-header.css`. Both old names existed only in Salt's legacy alias file and were undefined in `themeNext`, so the theme would silently render with undefined values under `SaltProviderNext`. After cleanup, every `--salt-*` reference in the package resolves to a token defined by both `theme.css` and `theme-next.css`, which means the v3 `withParams` mapping (§5) can be written directly against the modern token names without an intermediate rename step.
- **`scripts/convertClassName.mjs` deleted.** The four mode/density selectors are now checked into every CSS rule directly. The script is no longer needed, but the duplication itself remains — only v33's custom-property inheritance (§3.3) eliminates the *need* for the four copies, which is one of v3's killer wins.
- **Helper classes scoped to the Salt theme wrapper.** `.error-row`, `.warning-row`, `.success-row`, `.ag-column-drop-cell*` and `.ag-toggle-button-input-wrapper*` rules are now scoped to `[class*="ag-theme-salt"]` instead of bare class selectors, so they no longer collide with same-named consumer classes elsewhere on the page. In v3 the scoping concern disappears entirely because `saltCellStates`, `saltColumnDropStyle` and the toggle-button CSS only ship when their parts are loaded.

(Minor housekeeping in the same branch: `useAgGridHelpers` gets a `useRef<{ api: GridApi } | undefined>(undefined)` typing fix and drops a redundant `clsx(containerClassName, className)` call — no behaviour change. The hook is **removed** in v3, not replaced — see §9 decision 3.)

The migration plan in §7 remains unchanged; these fixes simply make Phases 1–3 strictly easier.

## 3. What changes in AG Grid v33+

### 3.1 The Theming API model

```ts
import { themeQuartz, colorSchemeDark, iconSetMaterial } from "ag-grid-community";

const myTheme = themeQuartz
  .withPart(iconSetMaterial)
  .withPart(colorSchemeDark)
  .withParams({ spacing: 12, accentColor: "red" });

<AgGridReact theme={myTheme} ... />
```

- **Themes are JS objects.** No more `import "...css"` and no required `class="ag-theme-*"` wrapper.
- The grid **injects its own CSS** into `<head>` (or wherever you point `themeStyleContainer`). The Theming API is module-aware, so you only ship CSS for features you use — smaller bundles.
- **`withParams(...)`** is the typed setter for ~150 named parameters (`accentColor`, `spacing`, `headerBackgroundColor`, `rowBorder`, …). Under the hood it just sets `--ag-*` CSS custom properties — but with TypeScript validation and an extended object syntax (`{ ref: "accentColor", mix: 0.5 }`, `{ calc: "4 * spacing - 2px" }`, `{ width: 1, style: "dashed", color: "..." }`, etc).
- **`withPart(part)`** swaps a feature like icons, checkboxes, inputs or tabs. Built-in parts exist for `colorScheme`, `iconSet`, `checkboxStyle`, `inputStyle`, `tabStyle`, `buttonStyle`, `columnDropStyle`.
- **`createTheme()`** starts from an empty theme containing only core grid styles — recommended when you'll replace most parts anyway (less bundle).
- **`createPart({ feature, params, css })`** lets organisations ship their own typed, named parts with scoped CSS.

### 3.2 Theme modes (light/dark)

When a theme uses `colorSchemeVariable`, the colour scheme switches at runtime via `data-ag-theme-mode="light|dark|dark-blue"` on **any parent element**. For Salt we don't strictly need this: `SaltProvider` already sets `data-mode` higher up and the `--salt-*` tokens it controls now inherit into the grid (§3.3). We expose `colorSchemeVariable` as an opt-in for non-Salt consumers — see §4.8.

### 3.3 Custom-property inheritance (big win)

> "CSS custom properties now inherit as expected — if you set `--ag-foreground-color` on the body element, it will be inherited by all grids on the page. In legacy themes, the custom property had to be set on the same element as the theme class." — v33 migration guide

This removes the *need* for the four hard-coded class selectors that the 2.x CSS sources still carry (one each for `light`, `dark`, `compact-light`, `compact-dark`). In v3 a single selector reads the inherited `--salt-*` tokens directly, regardless of which mode/density `SaltProvider` is in. Expect roughly a 75 % reduction in selector count once Phase 2 lands.

### 3.4 Spacing semantics changed

`--ag-grid-size` → `spacing`. The new `spacing` controls **padding** around elements (size = `max(iconSize, cellFontSize) + spacing-based padding`). Setting `spacing: 0` gives a no-padding grid; in legacy `--ag-grid-size: 0` gave zero-height rows. Salt's compact tier should be re-expressed via `spacing` + `rowVerticalPaddingScale` + `headerVerticalPaddingScale` rather than redefining `--salt-size-base` inside the grid.

### 3.5 Parameter renames / removals

A full mapping for our `ag-root-var.css` is in §5 below. Highlights:

- `--ag-data-color` → `cellTextColor`
- `--ag-header-foreground-color` → `headerTextColor`
- `--ag-control-panel-background-color` → `chromeBackgroundColor`
- `--ag-card-radius` → `borderRadius`
- `--ag-chip-background-color` → `columnDropCellBackgroundColor`
- `--ag-input-border-color` → `inputBorder` (composite border value)
- `--ag-input-focus-box-shadow` → `inputFocusShadow`
- `--ag-selected-tab-underline-*` → `tabSelectedUnderline*`
- `--ag-borders-secondary` + `--ag-header-column-separator-*` → unified `headerColumnBorder` + `headerColumnBorderHeight`
- `--ag-secondary-border-color`, `--ag-secondary-foreground-color`, `--ag-subheader-background-color`, `--ag-disabled-foreground-color` → **removed**, use targeted CSS or `borderColor`/`headerTextColor`
- `--ag-checkbox-*`, `--ag-toggle-button-*` → significantly reworked; use the new `checkboxStyle` part and the new `toggleButtonSwitchInset` parameter
- `--ag-icon-font-*` / `--ag-icon-image-*` → wrap in an `iconSet` part (`createPart({ feature: "iconSet", ... })`)
- `--ag-row-border-color/style/width` → unified `rowBorder`

### 3.6 Shadow DOM, CSS layers

- Shadow DOM is supported out of the box; the grid inserts styles inside the shadow root automatically. We should expose `themeStyleContainer` as a documented opt-in for consumers who need shared CSS across multiple grids in the same shadow root.
- CSS layers are first-class via the `themeCssLayer` grid option (`"grid"` wraps grid styles in `@layer grid { ... }`). This lets app stylesheets win over the grid without specificity wars — useful for Salt consumers integrating with their own design tokens.

### 3.7 Other quality-of-life wins

- Reduced selector specificity — `.ag-cell-inline-editing { box-shadow: none }` actually works now.
- Focus styles unified across header cells and form controls (`focusShadow`).
- Distributed themes / parts as TypeScript modules — consumers get autocomplete + inline docs for every Salt-specific parameter we expose.

### 4.2 Public API (target)

```ts
// @salt-ds/ag-grid-theme
import {
  saltTheme,            // ready-to-use theme (recommended default)
  createSaltTheme,      // factory if you want to override Salt defaults at build time
  saltAgGridDefaults,   // plain object: opt-in defaultColDef etc. (§9)
  // Parts (advanced — compose your own theme)
  saltIconSet,
  saltCheckboxStyle,
  saltInputStyle,
  saltColumnDropStyle,
  saltTabStyle,
  saltCellStates,
  saltFocusRing,
  saltRangeSelectionAdjustments,
  // Header parts (mutually exclusive within each feature group)
  saltHeaderPrimary, saltHeaderSecondary, saltHeaderTertiary,
  saltHeaderDividerPrimary, saltHeaderDividerSecondary, saltHeaderDividerTertiary, saltHeaderDividerNone,
  // Row variants (composable parts: feature "saltRowVariant" / "saltZebra")
  saltRowVariantSecondary,
  saltRowVariantTertiary,
  saltZebra,
} from "@salt-ds/ag-grid-theme";
```

Typical usage:

```tsx
import { saltTheme, saltAgGridDefaults } from "@salt-ds/ag-grid-theme";
import { SaltProvider } from "@salt-ds/core";

<SaltProvider mode="dark" density="medium">
  <AgGridReact theme={saltTheme} {...saltAgGridDefaults} rowData={...} columnDefs={...} />
</SaltProvider>
```

Variants:

```tsx
import { saltTheme, saltZebra, saltRowVariantSecondary } from "@salt-ds/ag-grid-theme";

const myTheme = saltTheme.withPart(saltRowVariantSecondary).withPart(saltZebra);
```

Sensible defaults are exposed as a **plain object — not a hook** — because there is nothing reactive to do (see §9, decision 3):

```ts
// src/saltAgGridDefaults.ts
import type { AgGridReactProps } from "ag-grid-react";

export const saltAgGridDefaults = {
  defaultColDef: { filter: true, resizable: true, sortable: true },
} satisfies Partial<AgGridReactProps>;
```

Consumers spread it (or skip it entirely): `<AgGridReact theme={saltTheme} {...saltAgGridDefaults} />`. Overrides are just additional props placed after the spread. Grid-instance management (`apiRef`, `isGridReady`, `sizeColumnsToFit` on ready) is generic AG Grid plumbing and stays in consumer code.

### 4.3 `saltTheme` skeleton (illustrative)

```ts
// src/saltTheme.ts
import { createTheme } from "ag-grid-community";
import { saltIconSet } from "./parts/saltIconSet";
import { saltCheckboxStyle } from "./parts/saltCheckboxStyle";
import { saltInputStyle } from "./parts/saltInputStyle";
import { saltColumnDropStyle } from "./parts/saltColumnDropStyle";
import { saltTabStyle } from "./parts/saltTabStyle";
import { saltCellStates } from "./parts/saltCellStates";
import { saltFocusRing } from "./parts/saltFocusRing";
import { saltRangeSelectionAdjustments } from "./parts/saltRangeSelectionAdjustments";
import { saltHeaderPrimary } from "./parts/header/saltHeaderPrimary";
import { saltHeaderDividerPrimary } from "./parts/header/saltHeaderDividerPrimary";

export const saltTheme = createTheme()
  // ---------- parts ----------
  // NOTE: `colorSchemeVariable` is deliberately NOT included by default — Salt's
  // `--salt-*` tokens already react to `SaltProvider`'s `data-mode` one level up
  // (see §4.8). Consumers who need AG's `data-ag-theme-mode` switch can opt in
  // with `saltTheme.withPart(colorSchemeVariable)`.
  .withPart(saltIconSet)
  .withPart(saltCheckboxStyle)
  .withPart(saltInputStyle)
  .withPart(saltColumnDropStyle)
  .withPart(saltTabStyle)                   // tabStyle part — owns tabSelectedUnderline{Color,Width} + bespoke tab CSS
  .withPart(saltHeaderPrimary)              // default; swappable via saltHeader{Secondary,Tertiary}
  .withPart(saltHeaderDividerPrimary)       // default; swappable via saltHeaderDivider{*}
  .withPart(saltCellStates)                 // app-level state classes (.error-cell, …)
  .withPart(saltFocusRing)                  // unified focus rings
  .withPart(saltRangeSelectionAdjustments)  // cross-cell outline, fill handle, etc.
  // ---------- parameters (the salt-tokens bridge) ----------
  .withParams({
    // sizing
    spacing:                    "var(--salt-spacing-50)",
    rowHeight:                  "calc(var(--salt-size-base) + var(--salt-spacing-100))",
    headerHeight:               "calc(var(--salt-size-base) + var(--salt-spacing-100))",
    listItemHeight:             "calc(var(--salt-size-base) + var(--salt-spacing-100))",
    // iconSize is owned by saltIconSet — do NOT redeclare here
    cellHorizontalPadding:      "var(--salt-spacing-100)",
    widgetContainerHorizontalPadding: "var(--salt-spacing-100)",
    rowGroupIndentSize:         { calc: "iconSize + spacing" },

    // typography
    fontFamily:                 "var(--salt-text-fontFamily)",
    fontSize:                   "var(--salt-text-fontSize)",
    headerFontWeight:           "var(--salt-text-label-fontWeight-strong)",

    // colours — key
    backgroundColor:            "var(--salt-container-primary-background)",
    foregroundColor:            "var(--salt-content-primary-foreground)",
    accentColor:                "var(--salt-actionable-accented-bold-background)",
    invalidColor:               "var(--salt-status-error-foreground-informative)",

    // colours — chrome / menus
    // (headerBackground/Hover/Text & headerRowBorder are set by the header parts above)
    chromeBackgroundColor:      "var(--salt-container-primary-background)",
    menuBackgroundColor:        "var(--salt-container-primary-background)",
    tooltipBackgroundColor:     "var(--salt-container-primary-background)",
    modalOverlayBackgroundColor:"var(--salt-overlayable-background)",

    // colours — rows
    cellTextColor:              "var(--salt-content-primary-foreground)",
    rowHoverColor:              "var(--salt-selectable-background-hover)",
    selectedRowBackgroundColor: "var(--salt-selectable-background-selected)",

    // colours — range selection
    rangeSelectionBackgroundColor: "var(--salt-overlayable-background-rangeSelection)",
    rangeSelectionHighlightColor:  "var(--salt-overlayable-background-highlight)",
    rangeSelectionBorderStyle:     "none",

    // borders (composite values, see Borders docs)
    borderColor:                "var(--salt-separable-secondary-borderColor)",
    rowBorder:                  { width: 1, color: "var(--salt-separable-tertiary-borderColor)" },
    columnBorder:               false,
    wrapperBorder:              false,   // matches `ag-root-var.css` lines 78–83 today
    headerColumnBorder:         { width: 1, color: "var(--salt-separable-tertiary-borderColor)" },
    headerColumnBorderHeight:   "calc(var(--salt-size-base) / 2 - 2px)",
    borderRadius:               "var(--salt-palette-corner)",

    // inputs
    inputBorder:                { color: "var(--salt-editable-borderColor)" },
    inputFocusBorder:           { color: "var(--salt-editable-borderColor-hover)" },
    inputFocusShadow:           "none",

    // tabs — see saltTabStyle part (§4.6.3) for tabSelectedUnderline{Color,Width}

    // overlays
    popupShadow:                "var(--salt-overlayable-shadow-modal)",

    // value change deltas
    valueChangeDeltaUpColor:    "var(--salt-sentiment-positive-foreground-decorative)",
    valueChangeDeltaDownColor:  "var(--salt-sentiment-negative-foreground-decorative)",
  });

// Optional factory if consumers want to override at build time:
export const createSaltTheme = (overrides?: Parameters<typeof saltTheme.withParams>[0]) =>
  overrides ? saltTheme.withParams(overrides) : saltTheme;
```

> **Type-check the params table.** Several entries above are CSS-string shortcuts (`spacing: "var(--salt-spacing-50)"`, `rowHeight: "calc(...)"`, …) that may need to be expressed via the typed `{ calc: "..." }` / `{ ref: "..." }` forms once they hit the published `ThemeParams` interface. Phase 0 (§7) treats "every entry compiles without `as any`" as an explicit exit criterion; expect to rewrite 10–20 % of the rows during the spike.
>
> **`iconSize` lives in `saltIconSet`** (§4.5), not the theme. Do not redeclare it — "last writer wins" is the kind of subtle bug we want to avoid.
>
> **Header params** (`headerBackgroundColor`, `headerCellHoverBackgroundColor`, `headerTextColor`, `headerRowBorder`) live in the header parts (§4.6.1), not in the theme defaults.

### 4.4 Salt-specific adjustment parts (the bespoke CSS)

The ~1,150 lines of `css/parts/*.css` fall into three buckets that map to three **separate** no-feature parts. We deliberately avoid one monolithic `saltAdjustments` because (a) consumers can opt out of any bucket via `withoutPart("…")` and (b) it forces us to keep new bespoke CSS small and categorised.

| Part | Covers | Source CSS today |
| --- | --- | --- |
| `saltCellStates` | Application-level cell/row state classes (`.editable-cell`, `.numeric-cell`, `.error-cell`, `.warning-cell`, `.success-cell`, `.error-row`, `.warning-row`, `.success-row`) plus disabled text states (former `--ag-disabled-foreground-color` consumers). The 2.x scoping (`[class*="ag-theme-salt"] .error-row`) becomes implicit — the part only loads when in use. | `ag-body.css`, `ag-row.css` (state selectors only) |
| `saltFocusRing` | Unified focus rings on header cells, cells, menus, tool panels | `ag-header.css`, `ag-menus.css`, `ag-tool-panel.css` (focus selectors) |
| `saltRangeSelectionAdjustments` | Cross-cell outline, range fill handle, floating-filter padding, header-cell menu-visible state | residual bits of `ag-header.css`, `ag-body.css` |

**Rule of thumb at PR review time:** anything expressible via a typed parameter (§3.1) must be a typed parameter. The adjustment parts are for selectors only — they ship CSS, never `params`.

```ts
// src/parts/saltCellStates.ts
import { createPart } from "ag-grid-community";
import css from "../css/salt-cell-states.css?inline";

export const saltCellStates = createPart({
  // No `feature` → can't be accidentally swapped out; removable via withoutPart("saltCellStates")
  css,
});

// src/parts/saltFocusRing.ts and src/parts/saltRangeSelectionAdjustments.ts follow the same shape.
```

All three are added to the default `saltTheme` in §4.3 so the out-of-the-box look is preserved. Anything not expressible as a typed param OR as one of these three categories needs a design discussion before it lands.

### 4.5 Icon set part

```ts
// src/parts/saltIconSet.ts
import { createPart } from "ag-grid-community";
import iconCss from "../css/salt-icons.css?inline";

export const saltIconSet = createPart({
  feature: "iconSet",                  // replaces any other icon set
  css: iconCss,                        // @font-face + all --ag-icon-font-code-* defs
  params: {
    iconFontFamily: "salt-icons",
    iconSize: "max(var(--salt-size-icon), 12px)",
    // ~50 codepoints: iconFontCodeAggregation, iconFontCodeArrows, ...
  },
});
```

The woff file ships in `dist/fonts/salt-icons.woff` and the CSS uses a relative `url(...)` that the consumer's bundler resolves (esbuild/Vite/webpack all handle this when the package is shipped as JS that re-exports the CSS string with the asset URL injected at build time).

### 4.6 Variants

Three exclusive variants today (secondary / tertiary / zebra) and they can be combined (`secondary` + `zebra`). Two options:

| Option | API | Pros | Cons |
| --- | --- | --- | --- |
| **A. Composable parts** | `saltTheme.withPart(saltRowVariantSecondary).withPart(saltZebra)` | Smaller API surface; mirrors AG Grid's part model | Need to lock the feature-group names up front to avoid misuse |
| **B. Pre-baked themes** | `saltThemeSecondary`, `saltThemeSecondaryZebra`, … | One-import simplicity; matches current class-name UX | Six exported themes; adding a future variant doubles the export count |

**Recommendation: A.** Lock the following feature-group keys so combinations behave predictably:

| Feature key | Parts (mutually exclusive within the group) | Default |
| --- | --- | --- |
| `saltRowVariant` | `saltRowVariantSecondary`, `saltRowVariantTertiary` | none (primary look) |
| `saltZebra` | `saltZebra` | off |
| `saltHeaderBackground` | `saltHeaderPrimary`, `saltHeaderSecondary`, `saltHeaderTertiary` | `saltHeaderPrimary` |
| `saltHeaderDivider` | `saltHeaderDividerPrimary`, `saltHeaderDividerSecondary`, `saltHeaderDividerTertiary`, `saltHeaderDividerNone` | `saltHeaderDividerPrimary` |

That gives four orthogonal axes that compose cleanly. Document the supported combinations in `usage.mdx`.

### 4.6.1 Header background & divider (parity with Salt `Table`)

The 2.x line ships `ag-theme-salt-header-{primary,secondary,tertiary}` and `ag-theme-salt-header-divider-{primary,secondary,tertiary,none}` modifier classes (via `css/parts/ag-header-variants.css` in `fix/ag-grid-theme-align-with-table` — see §2.1 and issue [#5391](https://github.com/jpmorganchase/salt-ds/issues/5391)) that mirror the Salt `Table` `<THead variant divider />` API.

In v3, replace these with typed parts so the same options surface through the
Theming API. The token mapping below is taken verbatim from the 2.x CSS:

```ts
// src/parts/header/*.ts
import { createPart } from "ag-grid-community";

// Header background — mutually exclusive `saltHeaderBackground` group
export const saltHeaderPrimary   = createPart({ feature: "saltHeaderBackground", params: { headerBackgroundColor: "var(--salt-container-primary-background)",   headerCellHoverBackgroundColor: "var(--salt-container-primary-background)"  } });
export const saltHeaderSecondary = createPart({ feature: "saltHeaderBackground", params: { headerBackgroundColor: "var(--salt-container-secondary-background)", headerCellHoverBackgroundColor: "var(--salt-container-secondary-background)" } });
export const saltHeaderTertiary  = createPart({ feature: "saltHeaderBackground", params: { headerBackgroundColor: "var(--salt-container-tertiary-background)",  headerCellHoverBackgroundColor: "var(--salt-container-tertiary-background)"  } });

// Header divider — mutually exclusive `saltHeaderDivider` group
export const saltHeaderDividerPrimary   = createPart({ feature: "saltHeaderDivider", params: { headerRowBorder: { color: "var(--salt-separable-primary-borderColor)" } } });
export const saltHeaderDividerSecondary = createPart({ feature: "saltHeaderDivider", params: { headerRowBorder: { color: "var(--salt-separable-secondary-borderColor)" } } });
export const saltHeaderDividerTertiary  = createPart({ feature: "saltHeaderDivider", params: { headerRowBorder: { color: "var(--salt-separable-tertiary-borderColor)" } } });
export const saltHeaderDividerNone      = createPart({ feature: "saltHeaderDivider", params: { headerRowBorder: false } });
```

Usage:

```ts
import { saltTheme, saltHeaderSecondary, saltHeaderDividerPrimary } from "@salt-ds/ag-grid-theme";

const myTheme = saltTheme
  .withPart(saltHeaderSecondary)
  .withPart(saltHeaderDividerPrimary);
```

Defaults baked into `saltTheme` remain primary background + primary divider, matching the 2.x defaults (and Salt `Table`'s defaults).

### 4.6.2 Column-drop / pill tokens (Salt `Pill` parity)

Issue [#6196](https://github.com/jpmorganchase/salt-ds/issues/6196): row group panel pills must use Salt `Pill`'s `--salt-actionable-bold-*` tokens. The 2.x fix (see §2.1) ships via `ag-column-drop-list.css`, which cascades a small set of `--ag-*` custom properties across the default / hover / active / `[aria-expanded="true"]` states. In v3 this becomes the `saltColumnDropStyle` part — note that AG Grid's `columnDropCell{Background,Border}Color` params only cover the resting state, so the hover/active variants still need a CSS layer:

```ts
// src/parts/saltColumnDropStyle.ts
import { createPart } from "ag-grid-community";
import css from "../css/salt-column-drop.css?inline";

export const saltColumnDropStyle = createPart({
  feature: "columnDropStyle",          // replaces AG's built-in columnDropStyle
  css,                                 // cascades --ag-* for :hover / :active / [aria-expanded]
  params: {
    columnDropCellBackgroundColor: "var(--salt-actionable-bold-background)",
    columnDropCellBorder:          { color: "var(--salt-actionable-bold-borderColor)" },
    columnDropCellTextColor:       "var(--salt-actionable-bold-foreground)",
  },
});
```

```css
/* src/css/salt-column-drop.css — drives the hover / active / expanded states */
.ag-column-drop-cell:hover {
  --ag-secondary-foreground-color: var(--salt-actionable-bold-foreground-hover);
  --ag-icon-font-color:            var(--salt-actionable-bold-foreground-hover);
  --ag-chip-background-color:      var(--salt-actionable-bold-background-hover);
}

.ag-column-drop-cell:active,
.ag-column-drop-cell[aria-expanded="true"] {
  --ag-secondary-foreground-color: var(--salt-actionable-bold-foreground-active);
  --ag-icon-font-color:            var(--salt-actionable-bold-foreground-active);
  --ag-chip-background-color:      var(--salt-actionable-bold-background-active);
}
```

The part ships in the default `saltTheme` in §4.3 so consumers get it automatically. (The CSS no longer needs `[class*="ag-theme-salt"]` scoping — the part is only loaded when in use.)

### 4.6.3 Tabs (selected-underline + bespoke tab CSS)

Salt's only meaningful tab customisation in 2.x is the selected-tab underline (driven by `--ag-selected-tab-underline-color/width` in `ag-root-var.css`) plus two small CSS rules in `ag-menus.css` for `.ag-tabs` padding and `.ag-tab` sizing. AG Grid's `tabSelectedUnderline*` params live on the `tab-style` part interface (not in `CoreParams`), so to type-check them — and to keep symmetry with the other Salt-owned parts (`saltIconSet`, `saltCheckboxStyle`, `saltInputStyle`, `saltColumnDropStyle`) — we ship a `saltTabStyle` part. (Decision recorded 2026-06-11; the alternative Option A using the built-in `tabStyleBase` was considered and rejected in favour of the part-encapsulation convention.)

```ts
// src/parts/saltTabStyle.ts
import { createPart } from "ag-grid-community";
import type { TabStyleParams } from "ag-grid-community";
import css from "../css/salt-tab.css?inline";

export const saltTabStyle = createPart<TabStyleParams>({
  feature: "tabStyle",                 // replaces AG's built-in tabStyle parts (Quartz / Material / Alpine / Rolodex / Base)
  css,                                 // .ag-tabs / .ag-tab layout rules carried over from ag-menus.css
  params: {
    tabSelectedUnderlineColor: "var(--salt-navigable-accent-indicator-active)",
    tabSelectedUnderlineWidth: "var(--salt-size-indicator)",
  },
});
```

```css
/* src/css/salt-tab.css — carries over the 2.x bespoke rules from css/parts/ag-menus.css */
.ag-tabs {
  padding: var(--salt-spacing-100);
}
.ag-tab {
  height: calc(var(--salt-size-base) + var(--salt-spacing-100));
  flex: 1 1 auto;
}
```

The part ships in the default `saltTheme` in §4.3. Consumers who want a different look swap it: `saltTheme.withPart(tabStyleQuartz)` (or `tabStyleMaterial`, `tabStyleAlpine`, `tabStyleRolodex`, `tabStyleBase`). Mutually exclusive via the shared `feature: "tabStyle"` key.

### 4.7 Density / compact tier

Today `compact` is a separate set of class names that override `--salt-size-base`. In the new model:

- Salt's density is set on `SaltProvider`. `--salt-size-base` already inherits into the grid wrapper in v33 (because custom properties inherit now).
- `rowHeight`, `headerHeight`, `listItemHeight` already reference `calc(var(--salt-size-base) + ...)`, so they react automatically.
- The legacy `useAgGridHelpers` JS branching on density to compute `rowHeight={25|37|49|61}` becomes **unnecessary** — drop it.
- A consumer wanting to override density can simply `saltTheme.withParams({ rowHeight: 21, headerHeight: 20 })` or use `rowVerticalPaddingScale: 0.75` if they want padding-relative compactness.

This is a meaningful UX improvement: density just works without extra props.

### 4.8 Mode (light/dark)

Two viable approaches; we recommend (A):

**A. Write through `--salt-*` (preferred).** Salt's `--salt-*` tokens already react to `SaltProvider`'s `data-mode` attribute one level up; because v33 lets custom properties inherit into the grid, the theme automatically resolves to the right colours without any AG-specific mode wiring. This is the simplest answer, removes a synchronisation point, and is why `colorSchemeVariable` is omitted from the default `saltTheme` (§4.3). For non-Salt consumers we document it as an opt-in: `saltTheme.withPart(colorSchemeVariable)`.

**B. Mirror `data-mode` → `data-ag-theme-mode`.** If we ever need explicit per-grid mode overrides (e.g. a dark grid inside a light page), ship a tiny, single-purpose component or hook inside `@salt-ds/ag-grid-theme` that copies the attribute — *not* a general-purpose wrapper hook (see §9 decision 3). **Do not** put this in `@salt-ds/core` — core should not take a soft dependency on AG Grid concepts.

Custom mode names remain supported via `.withParams({...}, "high-contrast")` if Salt ever ships an explicit high-contrast tier.

## 6. Backwards compatibility & rollout

### 6.1 Versioning strategy

| Branch | Package version | AG Grid support | Status |
| --- | --- | --- | --- |
| `legacy` (maintenance) | `@salt-ds/ag-grid-theme@2.x` (next minor lands the §2.1 changes from `fix/ag-grid-theme-align-with-table`) | v28 – v32, **and** v33+ with `provideGlobalGridOptions({ theme: "legacy" })` | Patch-only until the first stable AG Grid release that drops legacy themes, or six months after 3.0.0 GA, whichever comes first. Concrete EOL date published in the 3.0.0 changeset. |
| `main` | `@salt-ds/ag-grid-theme@3.0.0` | v33+ only | Active development |

Pre-release cadence:

- Cut `3.0.0-next.0` at the end of Phase 4.
- Hold stable `3.0.0` for **≥ 4 weeks** of `-next` adoption (the original 2-week estimate is too short for an enterprise grid theme).
- Promote to stable only after **≥ 1 partner team** has signed off on a real-app integration.

### 6.2 Breaking changes in 3.0.0

1. Package no longer ships a global CSS file (`salt-ag-theme.css`) — replaced by JS exports.
2. Peer dep raised to `ag-grid-community >= 33.0.0`.
3. `class="ag-theme-salt-*"` is no longer required (or recommended). It still works if the consumer also imports the legacy CSS, but we don't support that combination.
4. Variants (row and header) apply via `withPart()` instead of modifier classes. Consumers using the 2.x `ag-theme-salt-header-*` / `ag-theme-salt-header-divider-*` modifiers (§2.1) switch to `saltHeader{Secondary,Tertiary}` / `saltHeaderDivider{*}` parts.
5. The `useAgGridHelpers` hook is **removed**, not replaced. Sensible defaults are exposed via the plain-object export `saltAgGridDefaults`; consumers spread it into `<AgGridReact>` (see §4.2). Grid-instance management (`apiRef`, `isGridReady`, `sizeColumnsToFit` on ready) is generic AG Grid plumbing and belongs in consumer code, not the theme package.

### 6.3 Migration aid (manual)

The change is mechanical enough that a codemod is poor ROI for the size of the consumer footprint, and any `jscodeshift` pass that handles dynamic class names will be fragile. Ship the following instead:

- **A side-by-side migration guide** in `usage.mdx` with copy-pasteable before/after blocks for the five common shapes (default, with row variant, with header variant + divider, with compact density, with custom column defs).
- **An ESLint rule** (in `eslint-local-rules.js`) that flags `className=/ag-theme-salt/` and `import "@salt-ds/ag-grid-theme/salt-ag-theme.css"` as errors with autofix hints pointing at the migration guide.
- The legacy escape hatch (`provideGlobalGridOptions({ theme: "legacy" })` + stay on 2.x) documented as the official path for consumers who can't migrate yet.

### 6.4 Documentation updates

- `site/docs/components/ag-grid-theme/usage.mdx` — split into "Using v3 (Theming API)" and "Using v2 (legacy)".
- Add a migration section with side-by-side before/after blocks, including the new 2.x header variant modifiers → v3 header parts mapping.
- Update each `LivePreview` example to use `theme={saltTheme}`.

### 6.5 Visual regression

Re-baseline the Chromatic snapshots for every story under `packages/ag-grid-theme/stories/` (including the new `HeaderVariants` story added in §2.1). Use the existing `mode: "side-by-side"` rendering. Expect small pixel drift around focus rings and checkbox sizes (v33 changed both intentionally).

## 9. Decisions

| # | Question | Decision | Rationale |
| --- | --- | --- | --- |
| 1 | Variants as parts vs themes | **Parts** (§4.6), with locked feature-group keys: `saltRowVariant`, `saltZebra`, `saltHeaderBackground`, `saltHeaderDivider` | Composes cleanly; matches AG's part model; small API surface; aligns directly with the 2.x header modifiers shipping in §2.1 |
| 2 | Drop the explicit compact tier | **Yes** | Inheritance of `--salt-size-base` + `rowVerticalPaddingScale` covers it without JS branching (§4.7) |
| 3 | Ship a `useSaltAgGrid()` hook | **No.** Once theme + density move to the theme object there is nothing reactive for a hook to do — it would wrap a module-level constant. Instead export `saltTheme` plus a plain-object `saltAgGridDefaults` (§4.2) that consumers spread into `<AgGridReact>`. Grid-instance management (`apiRef`, `isGridReady`, `sizeColumnsToFit`) is generic AG Grid plumbing and stays in consumer code. | Smallest API that works; no false-positive hook-shaped indirection over a singleton |
| 4 | Mode wiring location | **Write through `--salt-*` directly** (§4.8 option A); expose `colorSchemeVariable` as opt-in for non-Salt consumers. If per-grid overrides are ever needed, mirror the attribute inside `@salt-ds/ag-grid-theme`, **not** `@salt-ds/core` | Avoids `core` taking a soft dependency on AG Grid concepts |
| 5 | Codemod | **No** — manual migration guide + ESLint rule (§6.3) | Mechanical migration; small consumer footprint; codemods on dynamic class names are fragile |
| 6 | Class-based shim in 3.0 | **No** | Clean break is clearer; legacy escape hatch via 2.x + `theme: "legacy"` covers slow movers |

### 9.1 Spike findings (Phase 0/1/2)

Empirical architecture decisions discovered while implementing the spike on
branch `feat/ag-grid-theme-v3-phase-0`. These extend (not contradict) the
strategic decisions in the table above; they're the "how", not the "what".

| # | Finding | Where it bites | Resolution |
| --- | --- | --- | --- |
| **A1** | **Compose-over-default** for typed-feature parts where Salt only opinionates on a few params (`saltCheckboxStyle`). The original §4.5 had every Salt part use `feature: "<x>Style"` to fully REPLACE the AG built-in. For checkbox that means reimplementing ~100 lines of shape-drawing CSS + SVG check/dash glyphs we don't need to maintain. | `saltCheckboxStyle` | Drop `feature: "checkboxStyle"` from the Salt part. Add `checkboxStyleDefault` to `saltTheme` immediately BEFORE the Salt overlay. AG merges params with later parts winning; the default's rendering CSS provides the shape, the Salt overlay provides colours + sizing CSS. |
| **A2** | **Replace-with-overlay** for parts where Salt opinionates on EVERYTHING and the AG default would over-style adjacent surfaces (`saltInputStyle`). Compose-over-default with `inputStyleUnderlined` / `inputStyleBase` draws a border on every ag-input — including floating filters and cell editors that 2.x leaves bare — because the theme-level `inputBorder` param feeds the AG default's CSS. | `saltInputStyle` | Keep `feature: "inputStyle"` to fully replace the AG default. Ship the bespoke 2.x CSS as the part's `css` payload (filter inputs, large-text editor, autocomplete popup). Add a baseline `.ag-input-field-input { border: none; outline: none; background: transparent }` reset that 2.x got for free from AG v32 but which AG v3 only ships via the built-in `inputStyle*` parts. Other inputs (cell editors, etc.) inherit browser-default + our reset — matches 2.x. |
| **A3** | **`@font-face` must be top-level CSS.** AG v3 wraps every part's `css` payload inside `:where(.ag-theme-<feature>-N) { ... }`. `@font-face` nested in a selector block is invalid CSS — browsers ignore it (no glyphs render). | `saltIconSet` | Inject the `@font-face` declaration at the document root via a `document.head.appendChild(<style>)` side-effect at module load. Idempotent guard on a fixed `id`; SSR-safe via `typeof document !== "undefined"`. Side-effect `import "../css/foo.css"` from the part file is unreliable because Vite tree-shakes it when the only consumer (`saltTheme`) doesn't reference the CSS module. |
| **A4** | **`:where(.ag-theme-part-N)` part wrapping has zero specificity** — when sibling stylesheets target `[class*=ag-theme-]` (specificity 0,1,0), they win the cascade regardless of document order. In Storybook this manifests as the legacy `ag-grid-community/styles/ag-grid.css` (imported by sibling 2.x stories) silently clobbering ~80 V3 theme params (header bg → transparent, font-size → 14px, etc.). | V3 stories in mixed-mode Storybook | `stories/v3/_setupV3.ts` removes the conflicting `<style data-vite-dev-id="...ag-grid.css">` tag at V3 story load, with a `MutationObserver` to re-remove on Vite HMR. Storybook-only — the production `@salt-ds/ag-grid-theme@3.x` package doesn't mutate consumer documents. |
| **A5** | **Storybook eagerly loads sibling story modules.** Top-level imports in `packages/ag-grid-theme/stories/ag-grid-theme.stories.tsx` (which side-effect-imports the legacy `ag-grid-community/styles/ag-grid.css`) execute even when the user navigates directly to a V3 story. Drives A4 — the legacy CSS lands in V3 iframes without the V3 story being any wiser. | All mixed-mode stories | Mitigated by A4. Documented in the proposal so future spikes don't waste hours debugging "why does V3 render OOTB". |
| **A6** | **AG v3's `.ag-cell-focus` is keyboard-gated.** AG only adds the focus class when the user navigates with Tab + arrow keys; JS `.focus()` and synthetic Playwright `click()` don't trigger it. Cell focus is keyboard-gated by design (clicks open editors, keys navigate). | Testing focus visually | Test focus paths via keyboard simulation, not pointer events. In Playwright that's `playwright-cli press Tab` then `press ArrowDown / ArrowRight`. Worth a footnote in any future visual-regression spec. |
| **A7** | **`columnDropCellBorderRadius` param emits a var, but `.ag-column-drop-cell` doesn't consume it.** AG v3 bakes a hardcoded `border-radius: 500px` (full pill shape) into the module CSS for column-drop pills. Setting the theme param produces `--ag-column-drop-cell-border-radius: 0px` correctly, but the rendered element ignores it. Same shape applies to `.ag-column-drop-cell { padding: calc(var(--ag-spacing) * 0.25) }`. | `saltColumnDropStyle` | Ship a CSS override in the part's `css` payload that re-reads the variable: `.ag-column-drop-cell { border-radius: var(--ag-column-drop-cell-border-radius, 0); padding: 0 var(--salt-spacing-50); }`. Worth checking for similar baked-in CSS in any new part. |
| **A8** | **`--ag-selected-row-border-color` has no typed v3 param.** Proposal §5 noted this. The 2.x `ag-root-var.css:47` set it via custom-property scope, and `.ag-row-selected` border-color + `::before` bar both consume it. In V3 with no AG default for the variable, the bar renders transparent. | `saltRangeSelectionAdjustments` | Set `--ag-selected-row-border-color: var(--salt-selectable-borderColor-selected)` on `.ag-root-wrapper` inside the part's CSS. Cascades to every descendant via custom-property inheritance. |
| **A9** | **`createPart<TabStyleParams>(...)` rejects partial params.** Passing the full `TabStyleParams` (19 keys) as the generic forces TS to require all 19. Salt only opinionates on 2. | `saltTabStyle` (and any future part touching a subset of a built-in params interface) | Narrow the generic to a `SaltTabStyleParams` interface containing only the keys Salt sets. The part still emits CSS that AG Grid consumes for the full feature; the type narrowing is purely a TS author-time concession. |
| **A10** | **`*.css?inline` needed a per-package TS declaration.** The repo-level `declarations.d.ts` had `*?inline` but not `*.css?inline` — the more specific glob doesn't match because `?inline` comes after `.css`. The package's `tsconfig.json` only includes `src/**/*`, so the repo-level declaration isn't visible to package-local imports anyway. | Every part that imports CSS via `?inline` | Added `packages/ag-grid-theme/src/declarations.d.ts` with both `*.css` and `*.css?inline` declarations. Same pattern as `@salt-ds/data-grid`. |
| **A11** | **Variant + zebra colour-flip doesn't translate to additive part merging.** 2.x flipped zebra colours when combined with `variant-secondary`/`variant-tertiary` (even rows go primary, odd go secondary). AG v3 part merging is last-writer-wins per param, not selector-based. | `saltZebra` + `saltRowVariant{Secondary,Tertiary}` combined | Phase 1 ships plain saltZebra standalone, matching what the legacy stories actually test (none combine variant + zebra). If a real consumer needs the combined look, ship dedicated combo parts (`saltRowVariantSecondaryZebra` etc.) in a future phase. Documented in `saltZebra.ts` JSDoc. |
| **A12** | **`headerTextColor` is repeated on every saltHeader{Primary,Secondary,Tertiary} part.** They share `feature: "saltHeaderBackground"` and are mutually exclusive; declaring `headerTextColor` only on Primary would lose the Salt grey when consumers swap to Secondary. | `saltHeader{Primary,Secondary,Tertiary}` | Three lines of duplication is fine for one param. If we add more header-related params in future phases, lift them into `saltTheme.withParams({...})` so they survive variant swaps. |

The 12-row table above is the empirical addendum to the strategic decisions in §9. Findings A1, A2, A7, A8, A11 in particular revise the original §4.5 / §4.6 / §5 wording — read them together with those sections when reviewing the v3 implementation.
