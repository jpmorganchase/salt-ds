Salt gives AI hosts a specialist workflow for applying the Salt design system inside real repos.

Salt's AI tooling is experimental, but the product model is stable:

- one public skill: `salt-ds`
- one workflow vocabulary: `init`, `create`, `review`, `migrate`, `upgrade`
- one preferred transport: MCP
- one fallback transport: CLI

For most consumers, the transport should stay in the background. Use Salt when you want design-system-aware help inside an existing codebase. Do not use Salt for generic React or CSS advice, freeform prompt-to-app generation, or guessed answers that ignore repo context.

## Golden Setup Path

Use this path first. Package READMEs are reference docs; this page is the consumer setup guide.

### 1. Install The Skill

Install the public `salt-ds` workflow skill:

```sh
npx skills add https://github.com/jpmorganchase/salt-ds/tree/main/packages/skills --list
npx skills add https://github.com/jpmorganchase/salt-ds/tree/main/packages/skills --skill salt-ds
```

These commands install the skill, not the CLI binary. Treat them as branch-tracking instructions rather than version-locked package installs.

If you are validating a local checkout or non-main branch, keep the skill, MCP, and CLI on the same checkout or branch instead of mixing a branch-pinned skill source with `@latest` packages.

### 2. Connect Salt MCP

If your host supports MCP, add the published Salt MCP package:

```json
{
  "mcpServers": {
    "Salt": {
      "command": "npx",
      "args": ["@salt-ds/mcp@latest"]
    }
  }
}
```

MCP is the preferred transport because it gives the host direct access to compact workflow tools, repo context, and retrieval follow-through.

### 3. Keep CLI As The Fallback

If MCP is blocked, stay on the same workflow model through the CLI:

```sh
npx -y @salt-ds/cli@latest info . --json
npx -y @salt-ds/cli@latest review src --json
```

Or install it once:

```sh
npm install -g @salt-ds/cli
```

After global install, use `salt-ds info`, `salt-ds create`, `salt-ds review`, `salt-ds migrate`, and `salt-ds upgrade` directly.

### 4. Verify The Setup

Run this first check in a consumer repo:

1. Run `npx -y @salt-ds/cli@latest info . --json` and confirm it returns repo context instead of a setup error.
2. Confirm the JSON includes `capabilityManifest` so hosts can inspect the compact contract version and workflow vocabulary without parsing docs.
3. If MCP is configured, ask the host to use `salt-ds` on an active Salt file.
4. Confirm the first answer is a Salt workflow response, not generic React or CSS advice.
5. Confirm the answer starts from compact workflow state and summary before raw detail.

### 5. Ask For One Clear Workflow

Start with one bounded job:

```text
Use salt-ds to review this file.
Focus on primitive choice, layout ownership, deprecated Salt usage, and the safest next fix.
```

Use `salt-ds init` only when you want durable `.salt` policy and the managed root instruction block after the first useful Salt result.

## Host Setup Cards

Use the same Salt workflow in every host. The host owns its MCP settings, local command execution, and natural-language explanation; Salt owns Salt evidence, workflow gates, repo policy, and unsupported states.
Use the first prompt in each card as the setup smoke check.

Expected consumer priority:

1. Copilot in VS Code or IntelliJ for most teams.
2. Claude Code for smaller advanced-user groups.
3. Cursor, Windsurf, Codex, or other MCP-capable hosts when a team already uses them.

- install the `salt-ds` skill from this branch or the published source

- keep repo-level Salt guidance in `AGENTS.md` when the repo has been initialized

## What Salt Is Good At

Salt is strongest when the job is clearly one of these:

- **Review**: review existing Salt UI, diagnose a Salt-specific bug, or validate recent implementation changes.
- **Upgrade**: separate required Salt version changes from optional cleanup.
- **Migrate**: replace non-Salt UI with Salt while preserving task flow, critical states, and interaction anchors.
- **Create**: add new Salt-native UI in an existing repo, especially bounded regions, screens, and workflow surfaces.
- **Init**: bootstrap repo policy and the managed instruction block after the first successful result.

`review` is often the best first trust loop, but choose the workflow by job, not by a fixed IDE order.

## Why Use Salt Instead Of A Generic Prompt

Salt is designed for questions that general-purpose coding agents usually handle poorly:

- which Salt primitive, pattern, or foundation should own this UI
- whether the repo already has policy that changes the correct answer
- whether the task is really `create`, `review`, `migrate`, or `upgrade`
- whether a migration preserved the important parts of the existing experience
- whether runtime evidence is actually needed or the source pass is enough

Salt should feel like design-system intelligence inside your host, not like a second product you have to learn.

## How Salt Works

Salt combines four layers in a fixed order:

1. canonical Salt guidance
2. detected repo context
3. declared repo policy
4. runtime evidence only when uncertainty remains

That means:

- **canonical Salt guidance** answers what Salt recommends
- **detected repo context** answers what this repo is using right now
- **declared repo policy** answers how this repo wants Salt applied
- **runtime evidence** answers what the rendered app is doing right now

Do not push repo-local wrappers or local patterns into the canonical Salt step.

## Generated Context And Direct Docs

Salt uses generated context first because agents need stable, machine-readable facts for workflow decisions. Generated registry and context artifacts are built from Salt sources and are better suited for entity resolution, package status, evidence refs, and workflow gates than a fresh prose-only docs fetch.

Direct docs are still useful. Use them for human-readable provenance, source freshness checks, and identifying docs or registry gaps that should be closed upstream. Do not use a direct docs fetch to bypass the Salt workflow, ignore compact `status` or `action.kind`, or treat unsupported Salt facts as resolved.

When MCP is available, prefer Salt workflow output and its evidence refs first. When MCP is unavailable, use the CLI fallback for the same workflow model. Fetch docs only as supporting context, and feed any confirmed new evidence back through Salt's workflow gates before implementing.

## What Good Salt Output Looks Like

A good Salt answer should:

- identify the job clearly
- stay grounded in canonical Salt guidance
- use repo context before making Salt-specific choices
- respect declared repo policy when it exists
- ask only the minimum blocking question when safe progress is unclear
- show the Salt summary before raw workflow details when available
- prefer compact workflow state first and use richer detail only when deeper artifacts are actually needed
- make the safest next step obvious

## Blocked Output Example

When Salt returns `blocked`, `partial`, `ask_user`, retrieve, or missing-evidence states, summarize the blocker from the compact fields. Do not add a new field; make the existing state obvious.

```text
Salt workflow: create
Blocked: profile page implementation is not safe for the exact requested UI yet.
Why: status=partial; action.kind=retrieve_entity; safety.blocking_reasons says Avatar still needs follow-through; evidence.missing includes "follow-through evidence for Avatar".
Smallest next action: call get_salt_entity for Avatar, then rerun create_salt_ui with resolved_entities before editing.
coding is allowed: no. Do not edit the blocked region until the rerun returns status=success, action.kind=implement, safety.exact_request_safe=true, and evidence.status=complete.
```

If `action.kind` is `ask_user`, ask the returned question and stop. Treat the user's answer as updated workflow input, not as permission to edit.

## Prompt Starters

### Review

```text
Use salt-ds to review this file.
Focus on primitive choice, layout ownership, deprecated Salt usage, and the safest next fix.
```

### Upgrade

```text
Use salt-ds to upgrade this feature from Salt 1.x to 2.x.
Separate required changes from optional cleanup and keep the scope to this feature folder.
```

### Migrate

```text
Use salt-ds to migrate this attached screen into Salt.
Preserve action order, empty-state behavior, and keyboard flow.
```

### Create

```text
Use salt-ds to add a dashboard summary area above this table.
Keep the current filters and table as the primary detail region.
```

### Explore Options

```text
Use salt-ds to show me two Salt-valid directions for this workflow surface.
Compare the safer default direction with one more opinionated direction, then explain the trade-offs.
```

## Common Surfaces

- **Dashboard or overview page**: keep the request page-level first; do not collapse the first ask into one metric card before the top-level direction is grounded.
- **Table plus filters**: focus on data review, filtering, selection, and follow-up actions.
- **Form page**: keep validation, status feedback, and action hierarchy clear.
- **Dialog workflow**: check primitive choice, action ordering, supporting content, and dismissal behavior.
- **Navigation shell**: preserve task flow and landmarks while moving the structure toward canonical Salt patterns.
- **Supporting states**: review loading, empty, error, and success states as part of the same workflow.

## Repo Policy

Use the workflow directly for first-run value.

Use `salt-ds init` when you want to add durable Salt policy and the managed root instruction block after the first successful result. For most teams, `.salt/team.json` is enough.

Recommended repo shape:

```text
consumer-app/
|-- AGENTS.md
`-- .salt/
    `-- team.json
```

Use `.salt/stack.json` only when the repo genuinely needs layering.

## Visual Inputs

Salt can help with attached screens and mockups through structured design evidence. This is the supported handoff lane for visual migration work:

1. the host inspects the screenshot, Figma frame, story, or mockup
2. the host normalizes what it saw into `migrate_visual_evidence_v1` or a smaller `source_outline`
3. Salt validates and uses the structured evidence in the migrate workflow
4. show the Salt summary first

Use [`migrate-visual-evidence-request.schema.json`](../../../packages/semantic-core/schemas/migrate-visual-evidence-request.schema.json) for adapter requests and [`migrate-visual-evidence-response.schema.json`](../../../packages/semantic-core/schemas/migrate-visual-evidence-response.schema.json) for normalized responses. The portable MCP/CLI fallback is still `source_outline` with `regions`, `actions`, `states`, and `notes`.

Keep uncertainty visible. If the host is not sure about a label, landmark, hierarchy, action order, disabled state, or screenshot-derived detail, put that in `notes` or `ambiguities` instead of silently turning it into Salt implementation guidance.

Salt MCP does not treat raw screenshots, image uploads, Figma frames, or design-tool payloads as first-class workflow inputs yet. Do not parse pixels in Salt MCP; let the host or adapter normalize visual evidence first.

## Runtime Evidence

Do not start with runtime inspection. Start with canonical Salt guidance and source validation.

Escalate to runtime evidence only when you still need confirmation about rendered structure, landmarks and accessible names, runtime errors, screenshots, or layout behavior.

Runtime evidence is evidence, not policy. Do not use runtime inspection to decide canonical Salt component or pattern choice.

When a Storybook story or app URL exists, the host should run validation outside Salt and feed the findings back into the Salt workflow:

1. run the available host tool, such as a Storybook preview, app URL check, Playwright test, axe check, browser inspection, or repo script
2. record the URL or command, relevant state, and concrete observation
3. classify each finding as a Salt design-system issue, generic runtime issue, generic accessibility issue, build/test issue, or missing evidence
4. pass only the Salt-relevant observations back into `review` or `migrate` as runtime evidence
5. rerun validation after edits and report the new result

The host owns command execution, browser control, screenshots, and test logs. Salt owns interpretation of Salt-specific evidence and should not become a Storybook, Playwright, axe, or app runner.

## Common Problems

**The agent picked the wrong workflow**

Be explicit about the job: `review`, `upgrade`, `migrate`, or `create`. Scope the request to one file, one region, one feature folder, or one screen when possible.

**The answer ignored repo wrappers or local rules**

Check these first:

1. `npx -y @salt-ds/cli@latest info . --json` or `salt-ds info . --json` if the CLI is installed globally
2. whether `.salt/team.json` exists
3. whether `.salt/stack.json` exists for advanced layering
4. whether the task actually recommends checking project conventions

If MCP project context is in play, also check:

- `resolution.status` is `resolved`
- `context_id` is non-null before reusing it on later repo-aware workflow calls
- `artifacts.summary.context_health.trusted` is `true`
- `artifacts.summary.retry_with.root_dir` is used when context came back as `fallback`, `needs_explicit_root`, or `mismatch`

**Create started coding too soon**

If a broad `create` result starts implementing custom sub-surfaces before it grounds the expected Salt patterns:

1. check whether the answer starts with the Salt summary
2. check whether follow-through is still required
3. complete required follow-through before coding unresolved sub-surfaces
4. ask blocking questions before guessing
5. run `salt-ds review` after the grounded first pass

If you are using the CLI artifact path for starter grounding, keep it explicit:

- use `salt-ds create "<query>" --starter-only --json`
- do not combine `--starter-only` with `--full`
- do not expect `starter-only` on `review`, `migrate`, or `upgrade`

**MCP is unavailable**

Stay on the same workflow model and use the CLI fallback.

## Package References

The package READMEs in the repo are reference docs, not alternate setup paths:

- `packages/mcp/README.md`: MCP server package, public tool surface, runtime metadata, and maintainer notes.
- `packages/cli/README.md`: CLI command reference and support/debug commands.
- `packages/skills/README.md`: skill authoring source and local validation notes.

## What Salt Does Not Do

Treat these as future or experimental rather than current product behavior:

- raw screenshot or image upload as a first-class MCP workflow input
- Figma-native or design-tool-native Salt generation
- general prompt-to-app generation
- visual-first `create` as the default product path

Salt should remain a specialist workflow product for real repos.
