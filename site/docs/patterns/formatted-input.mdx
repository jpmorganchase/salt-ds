## When to use
- When user input must follow a specific format
- When visual formatting improves data entry accuracy and readability
- When validation against a format pattern is required
- Common formats include phone numbers, card numbers, currency amounts, and postal/ZIP codes.

<Callout title="Depending on the use case, consider these existing Salt options:">
- **[Number input](/salt/components/number-input)**: Best for numeric values require increment/decrement controls or adjustments within a defined range.
- **[Date input](/salt/components/date-input)**: Best for single dates or date ranges.
- **[International phone number input (pattern)](/salt/patterns/international-phone-number-input)**: Best for country-specific phone number entry with a location selector.
</Callout>

## How to build

### Anatomy

Formatted input contains:
1. **Label:** Helps users understand what information to provide.
2. **Input field:** A single input field where users type their value.
3. **Helper text:** Assists users in providing the right information.

 <Diagram
    src="/img/patterns/formatted-input/formatted-input-anatomy.png"
    alt="The anatomy of the formatted input pattern"
    border
  />

### Layout
- Use [Input](/salt/components/input) to accept user input queries in text field.
- Use [Form Field](/salt/components/form-field) to wrap [Input](/salt/components/input) with visible label, helper text and a status message.

<Diagram
  src="/img/patterns/formatted-input/formatted-input-layout.png"
  alt="The layout of the formatted input pattern"
  border
/>

<Callout status="success" title="Best practices">
  - Wrap the input in [Form Field](/salt/components/form-field) to provide a visible label, help text, and a validation status message—supporting accessibility compliance. For more guidance, refer to the [W3C form instructions](https://www.w3.org/WAI/tutorials/forms/instructions/).
  - Follow [Input](/salt/components/input) and [Form Field](/salt/components/form-field) specifications for accessibility and consistency.
  - For instructions on formatting Form Fields, please refer to the [CIB Digital Content Style Guide](https://publisher.apps.test.na-5w.gap.jpmchase.net/cib-digital-content-style-guide/page/502).
</Callout>

### Formatting guidance

Use helper text and placeholder text to provide formatting guidance or an example of the expected format. If a placeholder is set, it appears in the input until the user enters a value.

<Diagram
  src="/img/patterns/formatted-input/formatted-input-guidance.png"
  alt="guidance of the formatted input pattern"
  border
/>

<Callout status="success" title="Best practices">
    Placeholder text should never be the only source of guidance. Since placeholder text disappears once the user begins typing and may not be announced by all screen readers, it is recommended to provide persistent formatting guidance through the label or helper text.
</Callout>

## Formatting behavior

The user types freely into the input and the value is formatted to a defined pattern when the user clicks away (or moves their focus) from the field.

<Callout>
    This pattern applies formatting on blur (after the user finishes and leaves the field) to keep interaction predictable and avoid unexpected changes during input, aligning with [WCAG 2.1 guidance](https://www.w3.org/TR/WCAG21) on predictable behavior for focus and input changes.
</Callout>

### 1. Before entry

- **Blank (no placeholder):** Field is empty with no visible mask; label/helper text provide guidance.
- **With placeholder text:** Field shows an example format as placeholder text until the user types.

<Diagram
  src="/img/patterns/formatted-input/formatted-input-before-entry.png"
  alt="Input before entry of the formatted input pattern"

/>

### 2. Entering a value

The user is actively entering or modifying the value.

<Diagram

  alt="Inout entering value of the formatted input pattern"
  border
/>

### 3. Submitting a value

After the user finishes typing and leaves the input, the system formats the value into its formatted way and shows it in the field.

<Diagram

### 4. Re-entering

When the user focuses a previously formatted field again, revert the input to the original value the user typed (the editing value). This supports a predictable editing experience by letting users continue from their last input without the system altering what they entered.

## Format validation

Validate user typed characters in Formatted Input by checking whether the entered value matches the expected pattern and formatting rules. This validation can be shown in real time while typing, on blur, or after submission, depending on the use case and severity of the rule.

For general validation behavior (error, warning, success states), refer to [Form Field validation guidance](/salt/components/form-field/examples#validation).

### Error

When the value does not meet the required format or criteria.
- Input turns to error state.
- An error message is displayed to guide the user.

<Diagram
  src="/img/patterns/formatted-input/formatted-input-error.png"
  alt="Error state of the formatted input pattern"
  border
/>

### Warning

When the value is technically correct but has potential issues.
- Input turns to warning state.
- An warning message is displayed to guide the user.

<Diagram
  src="/img/patterns/formatted-input/formatted-input-warning.png"
  alt="Warning state of the formatted input pattern"
  border
/>

### Success

Use a success state to reassure users when their input is correct. Avoid unnecessary success messages unless confirmation is important for the workflow.

When the value is valid
- No error or warning message is displayed.
- An optional success indicator can be displayed.

<Diagram
  src="/img/patterns/formatted-input/formatted-input-success.png"
  alt="Success state of the formatted input pattern"
  border
/>

For additional information on warning, error, and success messages, please refer to the [CIB Digital Content Style Guide](https://publisher.apps.test.na-5w.gap.jpmchase.net/cib-digital-content-style-guide/page/605).

## Invalid characters

If the user types a value that isn't compliant with the rules of the format pattern, show validation feedback while they are entering the value.

### 1. Default

Show the input with its normal helper text.

<Diagram
  src="/img/patterns/formatted-input/formatted-input-default.png"
  alt="Default state of the formatted input pattern"
  border
/>

### 2. User types

As the user enters a value, validate characters in real time.

<Diagram
  src="/img/patterns/formatted-input/formatted-input-user-types.png"
  alt="User types of the formatted input pattern"
  border
/>

### 3. Invalid character detected

Immediately show an error state. Replace the default helper text with a clear message explaining what’s allowed (e.g., “Only numbers and () - are allowed”).

<Diagram
  src="/img/patterns/formatted-input/formatted-input-invalid.png"
  alt="Invalid state of the formatted input pattern"
  border
/>

### 4-a. User corrects it

If the user removes the invalid character, dismiss the error and restore the original helper text.

<Diagram
  src="/img/patterns/formatted-input/formatted-input-correction.png"
  alt="Input correction of the formatted input pattern"
  border
/>

### 4-b. User does not correct it:

If the user leaves the field without correcting the value, keep the Error state on blur/submit until resolved.

<Diagram
  src="/img/patterns/formatted-input/formatted-input-error-correction.png"
  alt="Input without correction of the formatted input pattern"
  border
/>

<Callout status="success" title="Best practices">
    Reuse the same error message when possible for consistency; only change it if the blur/submit message adds clearer next steps without contradicting the real-time one.
</Callout>

## Input normalization

Input normalization is the practice of converting user-entered values into a consistent, expected representation—without forcing users to type in one exact format. The user should be able to enter the value in any format that is familiar to them, after which point the value is normalized to the preferred format of the system. This improves usability when the meaning of the value is clear, but the format varies (e.g., hyphens vs. spaces, parentheses, capitalization, leading/trailing whitespace).

<Diagram
  src="/img/patterns/formatted-input/formatted-input-normalization.png"
  alt="Input normalization of the formatted input pattern"
  border
  caption="Hyphens are allowed and will be normalized on blur"
/>

### Recommended behavior

- Accept flexible input while typing (don’t over-restrict keystrokes).
- Normalize on blur (or on submit) to a canonical format.
- Validate after normalization and provide clear feedback if rules are violated.

<Callout title="Defining normalization rules">
    Product teams should define normalization rules that match their domain, validation requirements, and downstream system needs.
</Callout>

## Dynamic preview

Dynamic preview shows the formatted result in real time as the user types. It helps users understand how formatting will be applied. While the user is typing, display a formatted preview of the value (without interrupting input).
- When the user leaves the field, the preview can disappear, since the input itself should now display the final formatted value
- The preview can appear near the label or next to the input, depending on the layout.

<Callout status="success" title="Best practices">
    - Place the dynamic preview near the label when there is enough space and it won’t compete with or crowd the label content.
    - If the label is likely to impede the preview, place the dynamic preview next to the input instead.
</Callout>

## Copy & paste behavior

Users can paste a value into the input field. Pasted input follows the same rules as typed input:
- Validate the pasted value (including invalid characters) and show feedback.
- Normalize the value as needed (typically on blur or submit).
- Format on blur so the field displays the preferred system format after the user finishes and leaves the input.
